# Emberpath

## Project type: phaser-game

## Purpose
Mobile-first top-down exploration game with story scenes, inspired by Pilgrim's Progress, using AI-generated art (Midjourney + Flow). An allegorical journey game POC that proves three things: character exploration, story scene triggers, and AI art rendering in-engine.

## Running

```bash
npm install
npm run dev      # Vite dev server at localhost:5173
npm run build    # Production build to dist/
```

### Editor (dev tool)

```bash
cd tools/editor
npm install
npm run dev      # Vite dev server at localhost:5174
npm run build    # Production build to tools/editor/dist/
```

## Controls

- **Desktop:** W/A/S/D to move up/left/down/right
- **Desktop dialogue choices:** Arrow keys browse (> prefix on selected), Enter confirms
- **Mobile:** Touch and drag anywhere to activate virtual joystick
- **Mobile dialogue choices:** Tap a choice to highlight it (browse), tap Confirm button to commit (browse-then-confirm pattern â€” prevents mis-taps)

### Dev / testing

- **`?reset=1`** URL param: on Title boot, wipes flags AND world save together (calls `resetWorld()`); the Title renders in the no-save state on the same frame; `history.replaceState` removes the param so a refresh does not re-trigger the wipe.
- **`?clearSave=1`** URL param: on Title boot, wipes ONLY the world save (flags persist); same one-shot semantics + URL cleanup. Useful when testing "Continue with a deleted save while my conversation flags persist."
- Both params combine: visiting `?reset=1&clearSave=1` performs both wipes; both `console.info` lines log; the URL cleans both params.

## Behavior rules

- **Scene flow:** TitleScene â†’ GameScene. StoryScene is a parallel scene launched/stopped by GameScene. TitleScene is always the first scene loaded.
- **Responsive scaling:** Phaser Scale.RESIZE mode â€” canvas adapts to container/viewport size. No fixed 800Ã—600. The tile map world is fixed at 1600Ã—1216 pixels; the camera viewport shows more or fewer tiles depending on screen dimensions.
- **Resize handling:** GameScene adds a direct `window.addEventListener('resize', ...)` that calls `this.scale.refresh()` after one `requestAnimationFrame` â€” this forces Phaser to re-detect parent dimensions on orientation change (Phaser's own handler only sets a dirty flag on window resize, which can miss DOM layout changes). The Phaser 'resize' event then propagates to all system handlers (dialogue, etc.) for repositioning. Listeners cleaned up on scene shutdown/destroy.
- **Movement:** Frame-based smooth movement (delta-time), not tile-snapping. Axis-independent collision allows sliding along walls and NPCs.
- **Collision:** Bounding box corners checked against tile map and NPC bounds. Out-of-bounds = collision.
- **Input priority:** Keyboard takes priority over joystick when any WASD key is held. Dialogue and StoryScene capture all input while active.
- **Virtual joystick:** Created on pointerdown, destroyed on pointerup. Uses scrollFactor(0) to stay fixed on screen.
- **Camera:** Dual-camera setup. Main camera follows player with zoom and world bounds. UI camera ('ui') has no zoom or scroll â€” renders dialogue, joystick, and thought bubbles at screen coordinates. Zoom = `min(viewportW, viewportH) / (TARGET_VISIBLE_TILES Ã— TILE_SIZE)`, clamped â‰¥ 1. On resize: zoom updates first, then bounds (order matters for clamp correctness), then centerOn player. UI objects must call `cameras.main.ignore(obj)` so the zoomed camera doesn't render them off-screen. World objects created after camera setup must call `uiCam.ignore(obj)` to prevent double-rendering.
- **NPC interaction:** Proximity-based (1.5 tile radius). Space (desktop) or tap (mobile) to interact. Interaction prompt appears/disappears based on distance.
- **Dialogue:** Bottom-screen text box with typewriter effect. Space/tap advances or completes typewriter. All positions derive from `scale.width`/`scale.height` â€” no hardcoded pixel values. Data-driven node graphs in `data/dialogues.ts`. Optional `portraitId` on `DialogueScript` (script-level default) and `DialogueNode` (per-node override) â€” when set, an NPC portrait `Image` renders anchored above the box's top-left (bottom edge flush with box top), sized `PORTRAIT_DESIGN_SIZE` (96 design px) on desktop and clamped to `canvasWidth * PORTRAIT_MOBILE_WIDTH_FRACTION` (0.22) on mobile so it doesn't overflow narrow viewports. When a portrait is active the speaker label x offsets to the right of the portrait (`s(BOX_PADDING) + portraitWidth + s(8)`); when no portrait is set, dialogue renders unchanged from the pre-portrait baseline (Whispering Stones script verifies this case). The portrait `Image` is created once per dialogue and texture-swapped on portrait-id change to avoid a one-frame flicker (Learning EP-01). **Node-level flag side-effects:** `DialogueNode.setFlags?: Record<string, FlagValue>` mirrors the `DialogueChoice.setFlags` shape â€” `DialogueSystem.showNode` fires `setFlag(k, v)` for every entry at the top of the method, BEFORE the typewriter starts, so a downstream `onFlagChange` subscriber registered before the dialogue starts sees the new value within the same call stack as `showNode` (US-72 Keeper rescue). **Chained story scene:** `DialogueScript.endStoryScene?: string` is captured at `start()` into a private field that survives the `close()` lifecycle (this.script is nulled BEFORE `onEndCallback` fires); `DialogueSystem.getEndStoryScene()` returns it during the `onEnd` callback. `GameScene.dialogueSystem.setOnEnd` reads `getEndStoryScene()` AFTER `flushSave()` and calls `launchStoryScene(id)` â€” the flush-first order means a tab-close mid-story-scene resumes with the dialogue's flag flips already committed. Existing scripts without `endStoryScene` close as before â€” no story scene launch.
- **Dialogue choices (desktop):** Arrow keys browse (> prefix), Enter or tap confirms immediately. No Confirm button on non-touch devices.
- **Dialogue choices (mobile):** Full-width 44px tappable rows. Tap to highlight (no commit). Confirm button appears below choices. Tap Confirm to commit. Browse-then-confirm pattern prevents mis-taps. Box expands dynamically to accommodate choices + Confirm.
- **Zone-level mutual exclusion (LEARNINGS #56):** When dialogue is active, movement, NPC interaction, trigger evaluation, and NPC awareness state transitions are all suppressed. NPCs in `dialogue` state hold their facing texture snapshotted at `enterDialogue` time and ignore awareness updates until `exitDialogue` fires. When StoryScene is active, GameScene is fully paused. Thought bubbles queue during dialogue and display after. On mobile, general taps are blocked while choice rows are active (`mobileChoicesActive` guard). The same suppression applies during `transitionInProgress` (area transitions, US-15), `spawnInProgress` (US-71 conditional NPC spawn fade), and `sharingInProgress` (US-85 ember-share pulse) â€” all four are early-returns at the top of `GameScene.update`, before the dialogue branch and the world-walk body.

- **Ember sharing (US-85):** The "Share warmth" verb is implemented as a dialogue choice with optional `firePulseTarget?: string` (NPC id) on `DialogueChoice`. When picked, `DialogueSystem.selectChoice` fires the choice callback BUT defers the auto-advance â€” the GameScene choice handler (`setOnChoice`) resolves the target via `npcSpritesById`, sets `sharingInProgress = true`, and invokes `EmberShareSystem.startPulse(player, npcSprite, onComplete)`. On pulse end, onComplete sets `npc_warmed_<targetId>` via `setFlag`, clears `sharingInProgress`, and calls `dialogueSystem.advanceAfterPulse(choice)` so the warming flag flip and the next dialogue node land on the same tick â€” downstream `onFlagChange` subscribers (light brightening via `LightingSystem.registerLight` idempotent overwrite, alpha-gate force-eval via `maybeUpdateAlphaGates(true)`) ride the same flag-flip tick. Refusal paths (e.g. Driftwood) and wary-decline paths (Old Man pre-Wren) DO NOT set `firePulseTarget` â€” no pulse plays, no warming flag set; the dialogue advances normally to a polite-decline node. **Per-choice condition** â€” `DialogueChoice.condition?: string` evaluates through `evaluateCondition` and is filtered at render time inside `DialogueSystem.showChoices` (the choice is hidden, not greyed out). The Share-warmth choice is gated `has_ember_mark == true AND npc_warmed_<id> == false` (or for Driftwood, `npc_refused_driftwood == false`) so it appears exactly when warming is possible. selectChoice indexes into `renderedChoices` (the post-filter list) so a hidden choice can never be picked even via index manipulation.
- **Trigger zones:** Fire on player entry (bounding box overlap). One-shot triggers use internal flags. Repeatable triggers require exit-then-re-enter. Conditions use flag comparisons with AND logic. Trigger types: `thought`, `story`, `dialogue`, `exit`.
- **Exit zones:** Defined in `ExitDefinition[]` on each area, separate from triggers. Walking into an exit zone fires fade-out â†’ scene restart with new area â†’ fade-in. Re-entrancy guard prevents double-fire during fade. Optional conditions (same syntax as triggers). Exit tiles rendered amber-gold (EXIT_COLOR 0xc89b3c). Non-existent destination logs console error without crashing.
- **Area transitions:** GameScene accepts `{ areaId, entryPoint }` via scene data. All systems (collision, movement, triggers, NPC interaction, dialogue, story scenes) are parameterized â€” they receive area data via constructor or method parameters, no global imports. During transition: movement, NPC interaction, triggers, and dialogue are suppressed (transitionInProgress guard).
- **Debug overlay:** F3 toggles visibility (off by default). Shows trigger zones as color-coded semi-transparent rectangles (blue=thought, magenta=story, green=dialogue, orange=exit), text labels with ID/type/condition, exit destination labels, NPC interaction radii as yellow circles. World-space at depth 50. Toggle is guarded during dialogue.
- **Flag persistence:** Flags stored in localStorage under 'emberpath_flags'. Flags work cross-area â€” set in one area, readable in another. Reset available from TitleScene. `resetAllFlags` is atomic â€” also calls `clearSave` from `saveState.ts`, so the existing reset call site wipes flags AND world save together. Once-per-session warn on `setItem` failure (parity with `saveState`). **homecoming-light flags (US-82, US-83, US-84, US-86):** `npc_warmed_wren` (true once Wren is warmed via the Share-warmth pulse), `npc_warmed_old_man` (true once the post-Wren Old Man warming pulse lands), `npc_refused_driftwood` (true after Driftwood politely declines the Ember â€” set by the choice's setFlags, NOT by a pulse onComplete), and `homecoming_complete` (set by the village-centre reflection trigger after both warmings). All four are wiped by `resetAllFlags` and the warming-flag onFlagChange subscribers re-run with value=undefined, restoring per-NPC light baselines and the cumulative-desat baseline on the same tick.
- **Conditional exits and decorations:** Both `ExitDefinition.condition` and `DecorationDefinition.condition` are evaluated through `systems/conditions.ts:evaluateCondition` â€” same parser, same flag store, no parallel logic. **Exit gating** is checked once per player-position update inside `GameScene.checkExitZones`: the transition is skipped (no fade, no scene restart) when the condition evaluates false; collision is NOT inferred from the gate â€” the gate purely controls the transition. The Ashen Isle east exit to briar-wilds (US-100) gates on `has_ember_mark == true` â€” pre-Ember the cell is unreachable anyway because conditional `bramble-cluster` ObjectInstances at the same cells block collision; the gate is defense-in-depth. **Decoration visibility** is set once at scene `create` from `evaluateCondition` and re-evaluated when ANY flag named in any decoration's condition changes â€” `subscribeToConditionalDecorations()` mirrors the conditional-object path so a mid-area flag flip (e.g. `has_ember_mark` flipping in Ashen Isle) updates visibility without a scene restart. The flag-change subscriber API on `triggers/flags.ts` fires per-change (not per-frame, Learning EP-01); GameScene tracks `conditionalDecorations` separately so unconditional decorations are never re-evaluated. **Conditional ObjectInstance** (US-100) extends the same pattern to objects: the bramble-cluster path uses ObjectInstances (not decorations) so the cells both block collision AND hide visually on flag flip â€” flag-flip-driven collision through `subscribeToConditionalObjects` + `rebuildObjectCollisionMap`. **Collision flips on flag change** without a scene restart â€” for `marsh_trapped` on Fog Marsh, conditional ObjectInstances flip impassability via the same path. The "no scene restart" rule means the player's frame-by-frame view sees collision and visibility update on the same frame as the trigger fired.

- **Ember warmth state (US-101):** Single-axis state owned by `EmberWarmthSystem`. Always constructed (owns ember_warmth flag load + Reset Progress reseed even on areas with no drain/quiet zones â€” cheap and side-effect-safe). Constants: `WARMTH_MAX = 1.0`, `WARMTH_FLOOR = 0.3` (hard floor â€” ember NEVER extinguishes; design promise to a 6-12 audience), `WARMTH_DRAIN_PER_SECOND = 0.08`, `WARMTH_RESTORE_PER_SECOND = 0.20` (restoration faster than drain â€” quiet places are *generous*, mirrors prayer), `WARMTH_WRITE_EPSILON = 0.005`. Flag persistence: `ember_warmth: number` in the existing `flags.ts` store; in-memory updates each frame, persisted only when `|current - lastPersisted| > epsilon` (Learning EP-01 â€” no per-frame localStorage IO). Load path scrub-on-corrupt: non-finite / non-number / out-of-range values seed back to `WARMTH_MAX` with once-per-session warn (mirrors `saveState` pattern). Reset Progress: `onFlagChange('ember_warmth')` subscriber receives `undefined` and seeds in-memory back to MAX on the same tick. Visual binding: GameScene's `update()` lerps the player ember overlay's `setRadius`/`setAlpha` and calls `LightingSystem.setPlayerWarmth(w)` â€” `getPlayerRadius()` lerps `playerLightRadius{Floor,Full}` post-Ember (pre-Ember falls back to `playerRadiusPre`). All four overlay constants and both light-radius constants live in `LIGHTING_CONFIG` as committed values â€” never undefined-by-default.

## Scaling tuning guide

How zoom and UI scaling work and where to tweak them.

### Camera zoom (how many tiles are visible)

`scenes/GameScene.ts` â€” `TARGET_VISIBLE_TILES` constant (default: 10). This controls how many tiles fit along the viewport's shorter axis. The zoom formula is `min(viewportW, viewportH) / (TARGET_VISIBLE_TILES Ã— TILE_SIZE)`, clamped â‰¥ 1.

- **Lower value** (e.g. 6) = more zoomed in, fewer tiles visible, characters/terrain appear larger. Good for close-up detail.
- **Higher value** (e.g. 14) = more zoomed out, more of the map visible, characters appear smaller. Good for wide-area overview.
- The `Math.max(1, ...)` clamp prevents zooming out below native resolution on large screens (desktop typically stays at zoom 1).

### Dialogue and UI text scaling

`systems/dialogue.ts` â€” all layout constants are in **design pixels** (authored for a reference viewport). The `s()` helper scales them by the main camera's zoom at render time: `s(v) = v * cameras.main.zoom`.

Tuning points (all in design pixels, top of dialogue.ts):
- `BOX_HEIGHT` (120) â€” dialogue panel height
- `BOX_PADDING` (12) â€” inner padding
- `SPEAKER_FONT_SIZE` (14) â€” NPC name label
- `TEXT_FONT_SIZE` (16) â€” dialogue body text
- `CHOICE_FONT_SIZE` (14) â€” choice row text
- `MOBILE_CHOICE_HEIGHT` (44) â€” tap target height per choice row
- `MOBILE_CONFIRM_HEIGHT` (44) â€” confirm button height
- `PORTRAIT_DESIGN_SIZE` (96) â€” square portrait render size on desktop (zoom 1)
- `PORTRAIT_MOBILE_WIDTH_FRACTION` (0.22) â€” mobile clamp as a fraction of `canvasWidth`

On desktop (zoom â‰ˆ 1), these render at their literal pixel sizes. On mobile (zoom â‰ˆ 3â€“4Ã—), they scale proportionally. To make mobile text larger without affecting desktop, increase the design-pixel values.

### Resize / orientation change

`GameScene.setupCamera()` registers a direct `window.resize` listener that calls `this.scale.refresh()` after one `requestAnimationFrame`. This forces Phaser to re-detect parent bounds on every resize/rotation, then Phaser emits its own 'resize' event which `handleResize()` (GameScene) and `handleResize()` (DialogueSystem) both listen to. No manual intervention needed â€” zoom, bounds, camera, and UI all recalculate automatically.
