# Emberpath

## Project type: phaser-game

## Purpose
Mobile-first top-down exploration game with story scenes, inspired by Pilgrim's Progress, using AI-generated art (Midjourney + Flow). An allegorical journey game POC that proves three things: character exploration, story scene triggers, and AI art rendering in-engine.

## Running

```bash
npm install
npm run dev      # Vite dev server at localhost:5173
npm run build    # Production build to dist/
```

### Editor (dev tool)

```bash
cd tools/editor
npm install
npm run dev      # Vite dev server at localhost:5174
npm run build    # Production build to tools/editor/dist/
```

## Controls

- **Desktop:** W/A/S/D to move up/left/down/right
- **Desktop dialogue choices:** Arrow keys browse (> prefix on selected), Enter confirms
- **Mobile:** Touch and drag anywhere to activate virtual joystick
- **Mobile dialogue choices:** Tap a choice to highlight it (browse), tap Confirm button to commit (browse-then-confirm pattern — prevents mis-taps)

### Dev / testing

- **`?reset=1`** URL param: on Title boot, wipes flags AND world save together (calls `resetWorld()`); the Title renders in the no-save state on the same frame; `history.replaceState` removes the param so a refresh does not re-trigger the wipe.
- **`?clearSave=1`** URL param: on Title boot, wipes ONLY the world save (flags persist); same one-shot semantics + URL cleanup. Useful when testing "Continue with a deleted save while my conversation flags persist."
- Both params combine: visiting `?reset=1&clearSave=1` performs both wipes; both `console.info` lines log; the URL cleans both params.

## Behavior rules

- **Scene flow:** TitleScene → GameScene. StoryScene is a parallel scene launched/stopped by GameScene. TitleScene is always the first scene loaded.
- **Responsive scaling:** Phaser Scale.RESIZE mode — canvas adapts to container/viewport size. No fixed 800×600. The tile map world is fixed at 1600×1216 pixels; the camera viewport shows more or fewer tiles depending on screen dimensions.
- **Resize handling:** GameScene adds a direct `window.addEventListener('resize', ...)` that calls `this.scale.refresh()` after one `requestAnimationFrame` — this forces Phaser to re-detect parent dimensions on orientation change (Phaser's own handler only sets a dirty flag on window resize, which can miss DOM layout changes). The Phaser 'resize' event then propagates to all system handlers (dialogue, etc.) for repositioning. Listeners cleaned up on scene shutdown/destroy.
- **Movement:** Frame-based smooth movement (delta-time), not tile-snapping. Axis-independent collision allows sliding along walls and NPCs.
- **Collision:** Bounding box corners checked against tile map and NPC bounds. Out-of-bounds = collision.
- **Input priority:** Keyboard takes priority over joystick when any WASD key is held. Dialogue and StoryScene capture all input while active.
- **Virtual joystick:** Created on pointerdown, destroyed on pointerup. Uses scrollFactor(0) to stay fixed on screen.
- **Camera:** Dual-camera setup. Main camera follows player with zoom and world bounds. UI camera ('ui') has no zoom or scroll — renders dialogue, joystick, and thought bubbles at screen coordinates. Zoom = `min(viewportW, viewportH) / (TARGET_VISIBLE_TILES × TILE_SIZE)`, clamped ≥ 1. On resize: zoom updates first, then bounds (order matters for clamp correctness), then centerOn player. UI objects must call `cameras.main.ignore(obj)` so the zoomed camera doesn't render them off-screen. World objects created after camera setup must call `uiCam.ignore(obj)` to prevent double-rendering.
- **NPC interaction:** Proximity-based (1.5 tile radius). Space (desktop) or tap (mobile) to interact. Interaction prompt appears/disappears based on distance.
- **Dialogue:** Bottom-screen text box with typewriter effect. Space/tap advances or completes typewriter. All positions derive from `scale.width`/`scale.height` — no hardcoded pixel values. Data-driven node graphs in `data/dialogues.ts`. Optional `portraitId` on `DialogueScript` (script-level default) and `DialogueNode` (per-node override) — when set, an NPC portrait `Image` renders anchored above the box's top-left (bottom edge flush with box top), sized `PORTRAIT_DESIGN_SIZE` (96 design px) on desktop and clamped to `canvasWidth * PORTRAIT_MOBILE_WIDTH_FRACTION` (0.22) on mobile so it doesn't overflow narrow viewports. When a portrait is active the speaker label x offsets to the right of the portrait (`s(BOX_PADDING) + portraitWidth + s(8)`); when no portrait is set, dialogue renders unchanged from the pre-portrait baseline (Whispering Stones script verifies this case). The portrait `Image` is created once per dialogue and texture-swapped on portrait-id change to avoid a one-frame flicker (Learning EP-01). **Node-level flag side-effects:** `DialogueNode.setFlags?: Record<string, FlagValue>` mirrors the `DialogueChoice.setFlags` shape — `DialogueSystem.showNode` fires `setFlag(k, v)` for every entry at the top of the method, BEFORE the typewriter starts, so a downstream `onFlagChange` subscriber registered before the dialogue starts sees the new value within the same call stack as `showNode` (US-72 Keeper rescue). **Chained story scene:** `DialogueScript.endStoryScene?: string` is captured at `start()` into a private field that survives the `close()` lifecycle (this.script is nulled BEFORE `onEndCallback` fires); `DialogueSystem.getEndStoryScene()` returns it during the `onEnd` callback. `GameScene.dialogueSystem.setOnEnd` reads `getEndStoryScene()` AFTER `flushSave()` and calls `launchStoryScene(id)` — the flush-first order means a tab-close mid-story-scene resumes with the dialogue's flag flips already committed. Existing scripts without `endStoryScene` close as before — no story scene launch.
- **Dialogue choices (desktop):** Arrow keys browse (> prefix), Enter or tap confirms immediately. No Confirm button on non-touch devices.
- **Dialogue choices (mobile):** Full-width 44px tappable rows. Tap to highlight (no commit). Confirm button appears below choices. Tap Confirm to commit. Browse-then-confirm pattern prevents mis-taps. Box expands dynamically to accommodate choices + Confirm.
- **Zone-level mutual exclusion (LEARNINGS #56):** When dialogue is active, movement, NPC interaction, trigger evaluation, and NPC awareness state transitions are all suppressed. NPCs in `dialogue` state hold their facing texture snapshotted at `enterDialogue` time and ignore awareness updates until `exitDialogue` fires. When StoryScene is active, GameScene is fully paused. Thought bubbles queue during dialogue and display after. On mobile, general taps are blocked while choice rows are active (`mobileChoicesActive` guard). The same suppression applies during `transitionInProgress` (area transitions, US-15), `spawnInProgress` (US-71 conditional NPC spawn fade), and `sharingInProgress` (US-85 ember-share pulse) — all four are early-returns at the top of `GameScene.update`, before the dialogue branch and the world-walk body.

- **Ember sharing (US-85):** The "Share warmth" verb is implemented as a dialogue choice with optional `firePulseTarget?: string` (NPC id) on `DialogueChoice`. When picked, `DialogueSystem.selectChoice` fires the choice callback BUT defers the auto-advance — the GameScene choice handler (`setOnChoice`) resolves the target via `npcSpritesById`, sets `sharingInProgress = true`, and invokes `EmberShareSystem.startPulse(player, npcSprite, onComplete)`. On pulse end, onComplete sets `npc_warmed_<targetId>` via `setFlag`, clears `sharingInProgress`, and calls `dialogueSystem.advanceAfterPulse(choice)` so the warming flag flip and the next dialogue node land on the same tick — downstream `onFlagChange` subscribers (light brightening via `LightingSystem.registerLight` idempotent overwrite, alpha-gate force-eval via `maybeUpdateAlphaGates(true)`) ride the same flag-flip tick. Refusal paths (e.g. Driftwood) and wary-decline paths (Old Man pre-Wren) DO NOT set `firePulseTarget` — no pulse plays, no warming flag set; the dialogue advances normally to a polite-decline node. **Per-choice condition** — `DialogueChoice.condition?: string` evaluates through `evaluateCondition` and is filtered at render time inside `DialogueSystem.showChoices` (the choice is hidden, not greyed out). The Share-warmth choice is gated `has_ember_mark == true AND npc_warmed_<id> == false` (or for Driftwood, `npc_refused_driftwood == false`) so it appears exactly when warming is possible. selectChoice indexes into `renderedChoices` (the post-filter list) so a hidden choice can never be picked even via index manipulation.
- **Trigger zones:** Fire on player entry (bounding box overlap). One-shot triggers use internal flags. Repeatable triggers require exit-then-re-enter. Conditions use flag comparisons with AND logic. Trigger types: `thought`, `story`, `dialogue`, `exit`.
- **Exit zones:** Defined in `ExitDefinition[]` on each area, separate from triggers. Walking into an exit zone fires fade-out → scene restart with new area → fade-in. Re-entrancy guard prevents double-fire during fade. Optional conditions (same syntax as triggers). Exit tiles rendered amber-gold (EXIT_COLOR 0xc89b3c). Non-existent destination logs console error without crashing.
- **Area transitions:** GameScene accepts `{ areaId, entryPoint }` via scene data. All systems (collision, movement, triggers, NPC interaction, dialogue, story scenes) are parameterized — they receive area data via constructor or method parameters, no global imports. During transition: movement, NPC interaction, triggers, and dialogue are suppressed (transitionInProgress guard).
- **Debug overlay:** F3 toggles visibility (off by default). Shows trigger zones as color-coded semi-transparent rectangles (blue=thought, magenta=story, green=dialogue, orange=exit), text labels with ID/type/condition, exit destination labels, NPC interaction radii as yellow circles. World-space at depth 50. Toggle is guarded during dialogue.
- **Flag persistence:** Flags stored in localStorage under 'emberpath_flags'. Flags work cross-area — set in one area, readable in another. Reset available from TitleScene. `resetAllFlags` is atomic — also calls `clearSave` from `saveState.ts`, so the existing reset call site wipes flags AND world save together. Once-per-session warn on `setItem` failure (parity with `saveState`). **homecoming-light flags (US-82, US-83, US-84, US-86):** `npc_warmed_wren` (true once Wren is warmed via the Share-warmth pulse), `npc_warmed_old_man` (true once the post-Wren Old Man warming pulse lands), `npc_refused_driftwood` (true after Driftwood politely declines the Ember — set by the choice's setFlags, NOT by a pulse onComplete), and `homecoming_complete` (set by the village-centre reflection trigger after both warmings). All four are wiped by `resetAllFlags` and the warming-flag onFlagChange subscribers re-run with value=undefined, restoring per-NPC light baselines and the cumulative-desat baseline on the same tick.
- **Conditional exits and decorations:** Both `ExitDefinition.condition` and `DecorationDefinition.condition` are evaluated through `systems/conditions.ts:evaluateCondition` — same parser, same flag store, no parallel logic. **Exit gating** is checked once per player-position update inside `GameScene.checkExitZones`: the transition is skipped (no fade, no scene restart) when the condition evaluates false; collision is NOT inferred from the gate — the gate purely controls the transition. **Decoration visibility** is set once at scene `create` from `evaluateCondition` and re-evaluated only when a referenced flag actually changes — the flag-change subscriber API on `triggers/flags.ts` fires per-change (not per-frame, Learning EP-01); GameScene tracks `conditionalDecorations` separately so unconditional decorations are never re-evaluated. **Collision flips on flag change** without a scene restart — for `marsh_trapped` on Fog Marsh, GameScene mutates `area.map` cells in place inside the same flag-change callback that re-evaluates decorations; `collidesWithWall` consumes `area.map` so the flip is automatic. The "no scene restart" rule means the player's frame-by-frame view sees collision and decoration update on the same frame as the threshold trigger fired.
- **Trigger flag side-effects:** `TriggerDefinition` carries optional `setFlags?: Record<string, FlagValue>` and `incrementFlags?: string[]`. On fire (via `TriggerZoneSystem.tryFire`), AFTER the one-shot bookkeeping and condition gate AND BEFORE the type-dispatch switch, the system iterates each field and calls `setFlag(key, value)` or `incrementFlag(name)`. The order is intentional — a downstream dialogue/story/thought callback reading the flag in the same frame sees the new value. `setFlags` mirrors the existing `DialogueChoice.setFlags` shape so authoring vocabulary stays consistent. Existing triggers without these fields fire identically (additive change).
- **Conditional NPC spawn:** `NpcDefinition.spawnCondition?: string` gates initial render and post-load spawn-on-flag-change. `GameScene.create` filters `area.npcs` by `evaluateCondition(spawnCondition)` into `activeNpcs`; subsystems (NpcBehaviorSystem, NpcInteractionSystem, moveWithCollision) receive `activeNpcs` so unspawned NPCs contribute zero collision/interaction/awareness. GameScene parses every spawn condition for flag names via `/\b([a-z_][a-z0-9_]*)\s*(==|!=|>=|>|<=|<)/gi`, deduplicates, and registers one `onFlagChange` listener per unique flag — `spawnConditionUnsubscribes[]` collected and invoked in `cleanupResize`. On any subscribed flag change, `evaluateConditionalSpawns()` walks the conditional NPC list once: any NPC whose condition is now true and is not yet in `npcSpritesById` is rendered (via shared `spawnNpcSprite`), pushed to `activeNpcs`, registered with NpcBehaviorSystem (via `registerNpc`), and faded in (`alpha: 0 → 1` over `KEEPER_FADE_DURATION_MS = 500`). Player input is suspended for `KEEPER_INPUT_SUSPEND_MS = 1000` via `spawnInProgress` (same zone-level-mutual-exclusion pattern as `transitionInProgress`). Idempotent — already-spawned NPCs are skipped via the `npcSpritesById` check; safely re-entrant from per-flag subscribers firing on the same frame.
- **Player ember overlay:** A small warm-gold disc (`Phaser.GameObjects.Arc`, `EMBER_COLOR = 0xf2c878`, radius 6) rendered at depth 5.5 above the player after the Keeper rescue (US-73). Created when `has_ember_mark` flips true via `onFlagChange`, AND on scene `create` if the flag is already true (so area transitions to Ashen Isle and Continue-from-save re-render the overlay without waiting for another flag flip). Destroyed when `has_ember_mark` is unset — `resetAllFlags` notifies subscribers with `undefined`, the subscriber tears the overlay down on the same frame as the flag wipe. Position update is a single `setPosition(player.x, player.y + EMBER_OFFSET_Y)` call at the end of `GameScene.update` — zero allocations, zero object literals (Learning EP-01). UI camera ignores the overlay (`uiCam.ignore`) so it does not double-render. Persists across area transitions (`scene.restart` re-runs `create`) and page reloads (the flag is in localStorage `emberpath_flags` per the existing flag-store contract, no new persistence layer).
- **Lighting:** A `LightingSystem` (`src/systems/lighting.ts`) renders a soft-falloff darkness overlay at depth 6 — between the Player ember overlay (5.5) and Thoughts (8). Implementation: a single `Phaser.GameObjects.RenderTexture` covering the world, filled solid black each frame, then "punched" by `rt.erase()` calls using a pre-baked radial-gradient brush texture (a single off-screen `Image` GameObject reused across all lights). The player is the primary light; radius/falloff selection switches between `playerRadiusPre/Post` based on the cached `has_ember_mark` flag. Configuration lives in `LIGHTING_CONFIG` (`src/systems/lightingConfig.ts`) and is live-tunable via F4 (toggles `LIGHTING_CONFIG.enabled`). With `enabled = false`, the overlay hides itself in one frame and the world renders identically to the keeper-rescue baseline (Rule 4a variant baseline). Per-frame `update()` body allocates zero JS objects — only `setPosition`/`setScale`/`setAlpha` mutations on the reused brush + `rt.fill`/`rt.erase` (Learning EP-01). Restart hygiene (Learning EP-02): all GameObject refs reset at top of `create()`; the brush canvas texture lives on the global texture manager and is guarded by `textures.exists` to avoid re-bake on `scene.restart`. POI/NPC/tier-2 lights register through `LightingSystem.registerLight({ id, x, y, radius, intensity, tier })` (US-75); tier-2 lights render at intensity 0 until `has_ember_mark === true`. The F3 debug HUD shows `enabled`, `playerRadius`, `desaturationStrength`, `hasEmber`, and `lights` count via the `DebugOverlaySystem.setHudProvider` hook.
- **POI lighting:** NPCs auto-register a tier-1 light at spawn time via `GameScene.registerNpcLight` — defaults to `LIGHTING_CONFIG.npcRadius` and `npcIntensity`. Per-NPC `lightOverride?: { radius?, intensity?, tier? }` lets authors tune individual NPCs (e.g. Old Man at intensity 0.15 reads as the Fading — visibly dimmer than other NPCs at the same distance). Triggers and decorations opt in via `light?: LightSpec` on their definition; `GameScene.registerInitialLights` registers them at scene `create` from `area.triggers` and `area.decorations`. Trigger lights are keyed `trigger:<id>`, decoration lights `decoration:<col>,<row>`, NPC lights are keyed by `npc.id`. Wandering NPCs carry their light: `LightingSystem.syncPositions(livePositions)` is called per frame after `npcBehavior.update`, mutating each light's x/y in place via `Map.get` lookup (Learning EP-01 — zero JS allocations in this path). Conditional NPC spawn (US-71) re-registers the light alongside the sprite via `registerNpcLight`. Tier-2 lights are registered alongside tier-1 but render at intensity 0 until `has_ember_mark === true` (the gate lives in `LightingSystem.update`, not at registration time, so the Ember flip is a single state change with no re-registration). Idempotent: `registerLight` with a duplicate id replaces the existing entry rather than duplicating (Learning #63).
- **Alpha-gated decorations (hidden-until-lit):** `DecorationDefinition.alphaGatedByLight?: boolean` (US-77) gates a decoration sprite's alpha on whether its centre falls inside any active light. `GameScene.renderDecorations` creates the sprite with `alpha: 0` (avoids a one-frame reveal flash on scene create) and tracks it in `alphaGatedDecorations[]` with its world-space centre. `maybeUpdateAlphaGates()` runs after the per-frame lighting update with two trigger conditions: (a) movement-gated — fires when the player has moved more than `LIGHTING_CONFIG.alphaGateRecheckPx` (default 8px) since the last check, and (b) forced — invoked from the `has_ember_mark` `onFlagChange` subscriber so the Ember flip atomically re-evaluates every gated decoration on the same frame as the radius/tier-2 reveal (no scene restart, no waiting for player movement). Coverage check is `LightingSystem.isLitAt(x, y)` — squared-distance comparison against the player's pre/post-Ember radius plus every registered light, with tier-2 lights skipped pre-Ember. Per-decoration cost is one isLitAt call only when the trigger fires; gated decorations registered without a paired tier-2 light will show as soon as the player walks within their own radius (intentional — author chooses by combining the `light` and `alphaGatedByLight` fields).
- **Desaturation post-FX:** A custom `DesaturationPipeline` (`src/systems/desaturationPipeline.ts`) is attached to the main camera in `GameScene.attachDesaturationPipeline`. It samples the LightingSystem's RenderTexture as a second sampler (texture unit 1; bound directly via `gl.activeTexture`/`gl.bindTexture` to avoid Phaser's PostFX wrapper API drift across versions). Per-fragment: maps the camera framebuffer UV into world UV using `cam.worldView` (passed each frame via `setFrameState`), samples the lighting mask alpha at that world UV, and mixes the original color toward luminance-weighted greyscale by `mask.alpha * currentStrength` — where `currentStrength` is a private pipeline field (NOT a direct read of `LIGHTING_CONFIG.desaturationStrength`) defaulting to `LIGHTING_CONFIG.desaturationStrength` and pushed by `GameScene.updateEffectiveDesaturation` whenever a warming flag changes (US-86 cumulative warming). The effective formula: `effective = max(DESAT_FLOOR, base × (1 − DESAT_REDUCTION_PER_WARMING × warmingsCount))` — base is `LIGHTING_CONFIG.desaturationStrength` (mutable for live-tuning); warmingsCount is the count of `true` flags across `WARMING_NPC_IDS`; reduction = 0.15 per warming; floor = 0.4 (preserves Ashen Isle's faded identity so the island never reads as fully restored). Recomputed ONLY on warming-flag-change events (and once at scene `create` for Continue-from-save resume), never per-frame (Learning EP-01). `cam.worldView` is the canonical "what's visible in world coords" — re-deriving from scroll/zoom/screen-size produced wrong results across viewports. `uFlipY` uniform handles Phaser's bottom-up framebuffer Y convention. With `LIGHTING_CONFIG.enabled = false` OR no lighting RT bound, the shader passes the framebuffer through unchanged (`uEnabled` short-circuit). The UI camera has no post-FX so dialogue/joystick/HUD render in original colors regardless of fog state. Thoughts (depth 8, main camera) render the cream/umber storybook bubble (US-88); their colors share the world's sepia palette so the desaturation pipeline tints them with the rest of the world by design — readability is preserved because the cream-vs-umber luminance contrast is high enough to survive luminance-weighted greyscale (no separate camera split needed). The F3 debug HUD reads the current effective strength via `desaturationPipeline.getStrength()` so the displayed value reflects post-warming reductions.
- **Save / resume:** Save layer is the world-walking checkpoint — `localStorage['emberpath_save']` holds `{ version: 1, areaId, position: { x, y } }`. Three write trigger points in `GameScene`: (1) **Throttled walk** — `maybeAutosave()` at the END of the world-walk branch in `update()`, gated by `SAVE_THROTTLE_MS` (1000ms) and `SAVE_MIN_DELTA_PX` (2px); guards return BEFORE any `JSON.stringify` so idle frames cost zero (Learning EP-01). (2) **Transition flush** — `transitionToArea` writes the DESTINATION payload BEFORE `scene.restart` so a tab-close mid-fade resumes on the destination's entry tile. (3) **Clean dialogue / StoryScene close** — `flushSave()` from the dialogue setOnEnd callback and from `events.on('resume', ...)` (StoryScene close path). Autosave is suppressed during transitions and dialogue (`zone-level mutual exclusion`, LEARNINGS #56); the flush-on-close paths re-arm autosave the moment the exclusion lifts. Title screen Continue precedence: `entryPoint` wins over `resumePosition` when both are set on `GameScene.create` boot data — transitions are deterministic, resume is opportunistic. URL-param resets fire on Title boot BEFORE layout reads `hasSave()`: `?reset=1` calls `resetWorld()`, `?clearSave=1` calls `clearSave()` only; both clean their consumed param via `history.replaceState`. Failures degrade gracefully — once-per-session warn on `localStorage.setItem` failure, otherwise silent.
- **Story scene rendering (US-89):** `StoryScene` is the parallel scene launched from `GameScene.launchStoryScene(id)` (or via `DialogueScript.endStoryScene` chain) that pauses GameScene and renders an illustrated cutscene over the screen. Each beat is a 256×256 PixelLab-generated PNG loaded from `assets/scenes/<sceneId>/beat-<n>.png` via `SCENE_ASSETS` (`art/sceneAssets.ts`); the texture key is `scene-<sceneId>-<beatIndex>` (or `scene-<assetRef>` when `StoryBeat.assetRef` overrides). On `advanceBeat`, the outgoing beat tweens α 1→0 and the incoming starts at α 0 and tweens to 1 over `BEAT_FADE_DURATION_MS = 400` — both coexist mid-tween, outgoing destroyed in `onComplete`. `crossFadeInProgress` flag blocks rapid double-tap re-entrancy after typewriter is complete (the existing two-tap typewriter-then-advance pattern is preserved; the new guard only applies once typewriter is complete and the cross-fade is in flight). Aspect-ratio preserved width-fit via `fitImageToSlot` with `STYLE_PALETTE.creamLight` letterbox at depth -1 — no squash on either viewport. Missing texture or `kind === 'animated'` falls back to flat-color rect + label without crashing (regression-safe). StoryScene's depth map is independent of GameScene's: -1 letterbox, 0 image (or fallback rect), 1 label, 10 panel, 11 beat text, 12 advance hint. Reset hygiene at top of `init()` + tween stop in `cleanupResize` (Learning EP-02). Aesthetic colors source from `STYLE_PALETTE` — no hardcoded panel/text/hint colors. Three scenes shipped: `ashen-isle-intro` (4 beats), `ember-given` (3 beats), `marsh-depths` (2 beats) — total 9 PixelLab generations.
- **Thought bubble rendering (US-88):** Cream/umber rounded panel for ambient narration, distinct from the bottom-screen dialogue UI (which has its own portrait + typewriter pipeline). Renders in world space on the main camera at depth 8, anchored to player position with `OFFSET_Y = -28` world-pixel offset above. Text uses `Phaser.GameObjects.Text` with a literal `fontSize: '12px'`, NO `fontFamily` (Phaser default sans-serif), `fontStyle: 'italic'`, and `color: STYLE_PALETTE.umberDark` — same approach as `systems/npcInteraction.ts`'s "Tap to talk" prompt so glyphs survive `pixelArt:true` NEAREST sampling at any camera zoom (Learning EP-04 — the earlier `dialogue.ts` `s(v) = v * mainCam.zoom` pattern was prescribed for this module but doesn't translate from UI-cam to main-cam rendering, and serif fontFamily was destroyed at mobile zoom; the fix matches a shipped main-cam pattern instead). Multi-line `wordWrap` with `useAdvancedWrap` (max 110 native px). Panel is `Phaser.GameObjects.Graphics.fillRoundedRect` with `STYLE_PALETTE.creamLight` fill α 0.92, 1px `STYLE_PALETTE.umberDark` stroke, radius 6. Symmetric vertical padding: panel drawn from `(-h + padY)` to `padY` around the text origin (0.5, 1) so `padY` sits above AND below text. UI camera ignores both `currentBg` and `currentText`. Reset hygiene at top of `displayThought` + `scene.events.shutdown` handler. Auto-dismiss after `DEFAULT_DURATION = 4000ms`; sequential queue waits for dialogue close.
- **Tile rendering (sprite-based):** Tiles are rendered via Phaser spritesheets loaded from `assets/tilesets/<id>/tilemap.png` (16×16 packed). Each `AreaDefinition` has a required `tileset: string` naming one of the registry entries in `maps/tilesets.ts`. `GameScene.preload()` loads every registered atlas. `renderTileMap()` iterates cells, determines tile kind (EXIT when cell overlaps an area.exits zone, else WALL or FLOOR from `area.map`), and calls `resolveFrame(tilesetId, kind, col, row)` — a pure function with FNV-1a hash determinism — to pick a variant. Sprites are placed at depth 0 and scaled to `TILE_SIZE` via `setDisplaySize`. Nearest-neighbor filtering is applied globally via `pixelArt: true` in the Phaser game config (`main.ts`). If `area.tileset` is unknown, a descriptive console error is logged and `renderFallbackTiles()` draws flat-color tiles using `visual.floorColor/wallColor` so the scene still loads. `EXIT_COLOR` and the old `tileGraphics` Graphics path are removed.
- **Decorative props (non-blocking):** `AreaDefinition.props: PropDefinition[]` is a list of `{ id, col, row, spriteFrame }` entries rendered by `GameScene.renderProps()` at depth 3 (between Tiles and Entities). Props look up their frame on the area's tileset atlas; a missing frame logs a warning naming the prop id and the missing frame, then skips that prop. Props are NOT added to collision structures — `collision.ts` is not consulted for props, so the player walks through their cells.
- **Decorations (tile-snapped, non-blocking):** `AreaDefinition.decorations: DecorationDefinition[]` is a list of `{ col, row, spriteFrame }` entries (no `id` — distinguishes them from props at the type level) rendered by `GameScene.renderDecorations()` at depth 2 (between Tiles and Props). Decorations are the authoring vocabulary used to compose visual structures from atlas frames — paths, building walls/roofs, fences, water/cliff edges, doorways, signs — without expanding the FLOOR/WALL collision enum. The `spriteFrame` is a literal atlas frame string; no `resolveFrame` variant picking, no hash-based randomness, fully deterministic. A missing frame logs a warning naming `(col, row)` and the missing frame, then skips that decoration. Decorations contribute zero to collision (the underlying base-layer FLOOR/WALL cell still controls walkability), and an out-of-bounds `(col, row)` renders off-grid harmlessly rather than crashing — authors see misplaced decorations on visual inspection. Created once at area load; no per-frame work (Learning EP-01).
- **Player sprite (fox-pip):** Player is a `Phaser.GameObjects.Sprite` at Entities depth 5, using PixelLab-generated fox child animation frames (96 PNGs — idle: 8 dirs × 4 frames = 32, walk: 8 dirs × 8 frames = 64). Frames preloaded individually in `GameScene.preload()` using per-animation-type frame counts (`FRAME_COUNTS: {idle: 4, walk: 8}`). 16 Phaser animations registered as `fox-pip-{idle,walk}-{north,north-east,east,south-east,south,south-west,west,north-west}` at 8 FPS with `repeat: -1`. Collision bounding box is PLAYER_SIZE (24px). Rendered at native scale (68×68px, `setScale(1)`) — collision math uses PLAYER_SIZE directly. Vite serves assets from `assets/` (`publicDir: 'assets'` in vite.config.ts).
- **Animation state machine (`systems/animation.ts`):** `AnimationSystem` manages sprite animation state. Two states: idle (stationary), walk (moving). Tracks `facingDirection` (default south on spawn), plays 8-direction animation. Octant math lives in `systems/direction.ts` — `velocityToDirection(vx, vy)` maps velocity to one of 8 directions using octant boundaries (atan2 quantized to 45° sectors): north, north-east, east, south-east, south, south-west, west, north-west. AnimationSystem imports it; NpcBehaviorSystem imports it — single source, no duplicate octant tables. `getCurrentSpeed()` always returns PLAYER_SPEED — no speed acceleration. GameScene passes raw input velocity directly (no diagonal suppression).
- **NPC sprite rendering:** NPCs render as `Phaser.GameObjects.Sprite` at depth 5 (Entities layer). `GameScene.preload()` iterates the `NPC_SPRITES` registry (`systems/npcSprites.ts`) and loads per sprite id: idle frames (8 dirs × 4), walk frames (8 dirs × 4), and 8 single-image static poses. `registerAnimations()` creates `npc-{spriteId}-{idle,walk}-{dir}` Phaser animations at 8 fps `repeat: -1`. Static poses are NOT registered as animations — they are loaded as plain image keys `npc-{spriteId}-static-{dir}` and applied via `setTexture`. Adding a new NPC is a registry entry plus an `AreaDefinition.npcs` row — no GameScene code edit. Unknown sprite id falls back with a descriptive console error to a lowercase rectangle draw so the scene still loads (silent failure is not acceptable).
- **NPC behaviour (`systems/npcBehavior.ts`):** Each NPC runs a per-frame state machine with four states — `idle`, `walk`, `aware`, `dialogue`. Wander: from the NPC's current tile a random direction is picked uniformly and Chebyshev-clamped to `wanderRadius` around the spawn tile; the path from current centre to candidate tile centre is sampled 4× and the step is rejected if any sample collides with a wall. Dwell time is randomised per cycle in `[NPC_IDLE_MIN_MS, NPC_IDLE_MAX_MS]`. Awareness: the system computes Chebyshev tile distance to the player every frame; on entry (idle/walk → aware) the NPC halts, on exit (aware → idle) it resets the dwell timer. While aware, the static texture is reapplied ONLY when the quantised facing direction changes — `lastStaticDir` guard prevents per-frame `setTexture` with identical inputs (Learning EP-01 loop-invariant). Dialogue: `enterDialogue(npcId, playerCenter)` is idempotent, snaps once to face the player, stops the animation, applies the static texture; `exitDialogue(npcId)` is idempotent and transitions back to idle so awareness re-evaluates on the next tick. `getLivePositions()` exposes centre-of-bounding-box coords for `collision.ts` + `systems/npcInteraction.ts`; both fall back to the static spawn tile when no live snapshot is available (pre-first-update window after an area transition).
- **Debug overlay NPC radii (F3):** interaction radius (solid yellow) → wander radius (dashed green) → awareness radius (dashed yellow), drawn in that order at depth 50. The order is the deterministic visual z-order (Learning #57 facet) and documented inline in `debugOverlay.ts`.

## Scaling tuning guide

How zoom and UI scaling work and where to tweak them.

### Camera zoom (how many tiles are visible)

`scenes/GameScene.ts` — `TARGET_VISIBLE_TILES` constant (default: 10). This controls how many tiles fit along the viewport's shorter axis. The zoom formula is `min(viewportW, viewportH) / (TARGET_VISIBLE_TILES × TILE_SIZE)`, clamped ≥ 1.

- **Lower value** (e.g. 6) = more zoomed in, fewer tiles visible, characters/terrain appear larger. Good for close-up detail.
- **Higher value** (e.g. 14) = more zoomed out, more of the map visible, characters appear smaller. Good for wide-area overview.
- The `Math.max(1, ...)` clamp prevents zooming out below native resolution on large screens (desktop typically stays at zoom 1).

### Dialogue and UI text scaling

`systems/dialogue.ts` — all layout constants are in **design pixels** (authored for a reference viewport). The `s()` helper scales them by the main camera's zoom at render time: `s(v) = v * cameras.main.zoom`.

Tuning points (all in design pixels, top of dialogue.ts):
- `BOX_HEIGHT` (120) — dialogue panel height
- `BOX_PADDING` (12) — inner padding
- `SPEAKER_FONT_SIZE` (14) — NPC name label
- `TEXT_FONT_SIZE` (16) — dialogue body text
- `CHOICE_FONT_SIZE` (14) — choice row text
- `MOBILE_CHOICE_HEIGHT` (44) — tap target height per choice row
- `MOBILE_CONFIRM_HEIGHT` (44) — confirm button height
- `PORTRAIT_DESIGN_SIZE` (96) — square portrait render size on desktop (zoom 1)
- `PORTRAIT_MOBILE_WIDTH_FRACTION` (0.22) — mobile clamp as a fraction of `canvasWidth`

On desktop (zoom ≈ 1), these render at their literal pixel sizes. On mobile (zoom ≈ 3–4×), they scale proportionally. To make mobile text larger without affecting desktop, increase the design-pixel values.

### Resize / orientation change

`GameScene.setupCamera()` registers a direct `window.resize` listener that calls `this.scale.refresh()` after one `requestAnimationFrame`. This forces Phaser to re-detect parent bounds on every resize/rotation, then Phaser emits its own 'resize' event which `handleResize()` (GameScene) and `handleResize()` (DialogueSystem) both listen to. No manual intervention needed — zoom, bounds, camera, and UI all recalculate automatically.
